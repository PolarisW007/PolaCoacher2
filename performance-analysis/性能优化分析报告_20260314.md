## 二、问题根因分析

### 核心问题："读书页返回后页面卡死"

**直接原因：**

1. **Audio 事件监听器泄漏（最核心）**：离开 study/player 页面时，`timeupdate` 回调仍以高频率（约每 250ms）持续触发 `setState`，React 试图更新一个已卸载的组件树，导致浏览器 JS 线程被密集的虚拟 DOM diff 操作阻塞。
2. **多个 setInterval 未清理**：share 轮询、completion 轮询、doc 轮询等在组件卸载后继续执行，叠加 setState 调用。
3. **缺少统一的 unmount 保护**：几乎所有 async 操作（fetch、Audio 事件、setTimeout）完成后都直接 setState，未检查组件是否仍挂载。

这些问题叠加在一起，在用户离开页面后会产生**大量无效的 setState 调用**，最终导致浏览器 JS 主线程被阻塞，页面表现为"卡死"。

### 修复核心技术方案

1. **audioListenersRef 模式**：使用 `ref` 数组保存所有通过 `addEventListener` 添加的监听器引用，在 `stopAudio()` 和 unmount cleanup 中统一 `removeEventListener`，彻底杜绝监听器泄漏。

2. **isMountedRef 模式**：在组件 mount 时设为 `true`，unmount 时设为 `false`。所有异步回调（fetch、setTimeout、setInterval、Audio 事件）在调用 `setState` 前检查此标志。

3. **autoPlayTimeoutRef 模式**：将 `setTimeout` 返回的 ID 保存到 ref，在 unmount 和 stopAudio 时 `clearTimeout`，防止延迟回调在组件卸载后执行。

4. **OrderedDict + LRU 淘汰**：后端 `_audio_hash_cache` 从普通 `dict` 改为 `collections.OrderedDict`，每次写入时 `move_to_end`，超过 10000 条时 `popitem(last=False)` 淘汰最久未用的条目。
